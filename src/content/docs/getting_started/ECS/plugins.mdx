Plugins are not part of a standard ECS architecture. In WasserXR, plugins are a
way of bundling components, systems and asset types into **reusable modules**.
These modules can then be seperately compiled from other parts of your project.
Unlike standard ECS implementations, WasserXR uses a plugin system to enable one
of its most powerful features: **hot-reloading**.

A plugin is simply a shared library (`.so` file on Linux, `.dll` on Windows)
that contains one or more components, systems or asset types.

## Creating Your First Plugin

<Tabs syncKey="lang">
<TabItem label="Rust">

To implement a plugin you have two options. Either you implement it directly in
the `lib.rs` of your main crate or you make a seperate crate for it. Making a
seperate crate makes sense, when you want to publish your components or systems
seperately.

Regardless, to make rust build dynamic libraries you have to add the following
to your `Cargo.toml` file.

```toml
[lib]
crate-type = ["cdylib", "staticlib", "rlib"]
```

This will tell rust to build library as a rust crate, static library (`.a`) and
as a dynamic library (`.so`)

Furthermore, add the following linker options to your `build.rs` file to fully
export all your symbols to the ABI:

```rust
fn main() {
    if std::env::var("CARGO_CFG_TARGET_OS").as_deref() == Ok("linux") {
        println!("cargo::rustc-link-arg=-Wl,--export-dynamic");
    }
}
```

</TabItem>
</Tabs>

## Loading and Unloading Plugins

<Tabs syncKey="lang">
<TabItem label="Rust">

To load and unload a plugin you can use the following functions

```rust
let _ = scene.load_plugin("./path/to/plugin.so".to_owned());

let _ = scene.unload_plugin("./path/to/plugin.so");
```

</TabItem>
</Tabs>

<Aside type="caution" title="Unloading a plugin" icon="warning">

When unloading a plugin, this will destroy all components, systems and assets
coming from the plugin.

</Aside>

## Reloading Plugins

This is the most special feature of WasserXR. When you change your code, you can
**recompile your plugin** and reload it **without restarting your application**:

<Tabs syncKey="lang">
<TabItem label="Rust">

In rust, you can trigger a reload by running

```rust
let _ = scene.reload();
```

</TabItem>
</Tabs>

## Dynamic and Static Linking

WasserXR supports both dynamic and static linking for your plugins:

|                                           | Dynamic Linking        | Static Linking |     |     |
| ----------------------------------------- | ---------------------- | -------------- | --- | --- |
| Purpose                                   | Development or Modding | Release        |     |     |
| Needs Plugin Loading                      | ✅️                    | ❌️            |     |     |
| Allows Hotreloading                       | ✅️                    | ❌️            |     |     |
| Needs the Plugins on end system installed | ✅️                    | ❌️            |     |     |

<Aside type="note" title="Hybrid Approach" icon="information">
  You can mix both approaches: statically link core functionality and
  dynamically load optional features. This allows also for direct **modding
  support**. The engine treats them the same way once loaded.
</Aside>

### How to do static linking

<Tabs syncKey="lang">
<TabItem label="Rust">

In rust, it is recommended to use the following setup, when you would like to
have a create development and release setup and you are working with other rust
plugins.

Add to your `Cargo.toml` the following

```toml
[dependencies]
<your_plugin> = {version = "<version>", optional = true}

[features]
default = []
static = ["dep:wasserxr-core"]
```

Then in your `main.rs` you add the following to ensure that all the symbols are
properly accessible in the end binary

```rust
// At the top of your file
#[cfg(feature = "static")]
use <your_plugin> as _;

// ...

// Where you load the plugins in your `main` function
#[cfg(not(feature = "static"))]
{
  scene.load_plugin("<path_to_dynamic_library>".to_owned()).expect("Failed to find plugin");
}
```

With this setup, when you run

```bash
cargo run
```

It will automatically use dynamic linking for development.

When you run

```bash
cargo run --release --features static
```

Then it will statically link everything together and make a clean release build.

</TabItem>
</Tabs>
