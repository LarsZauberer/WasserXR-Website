Plugins are not part of a standard ECS architecture. In WasserXR, plugins are a
way of bundling components, systems, and asset types into **reusable modules**.
These modules can be compiled separately from other parts of your project.
WasserXR uses plugins to enable one of its most powerful features:
**hot-reloading**.

A plugin is simply a shared library, such as a `.so` file on Linux, that
contains one or more components, systems, or asset types.

## Creating Your First Plugin

<Tabs syncKey="lang">
<TabItem label="Rust">

To implement a plugin, you have two options. Either you implement it directly in
the `lib.rs` of your main crate, or you make a separate crate for it. A separate
crate makes sense when you want to publish your components or systems
separately.

To make Rust build a dynamic library, add the following to your `Cargo.toml`
file:

```toml
[lib]
crate-type = ["cdylib", "staticlib", "rlib"]
```

This tells Rust to build the plugin as a normal Rust library, a static library
(`.a`), and a dynamic library (`.so`).

Furthermore, add the following linker options to your `build.rs` file to fully
export all your symbols to the ABI:

```rust
fn main() {
    if std::env::var("CARGO_CFG_TARGET_OS").as_deref() == Ok("linux") {
        println!("cargo::rustc-link-arg=-Wl,--export-dynamic");
    }
}
```

</TabItem>
</Tabs>

## Loading and Unloading Plugins

<Tabs syncKey="lang">
<TabItem label="Rust">

To load and unload a plugin, use the following functions:

```rust
let _ = scene.load_plugin("./path/to/plugin.so".to_owned());

let _ = scene.unload_plugin("./path/to/plugin.so");
```

</TabItem>
</Tabs>

<Aside type="caution" title="Unloading a plugin" icon="warning">

Unloading a plugin removes all systems, asset types, assets, and components
that came from that plugin.

</Aside>

## Reloading Plugins

This is the most special feature of WasserXR. When you change your code, you can
**recompile your plugin** and reload it **without restarting your application**:

<Tabs syncKey="lang">
<TabItem label="Rust">

In Rust, you can trigger a reload by running:

```rust
let _ = scene.reload();
```

</TabItem>
</Tabs>

## Dynamic and Static Linking

WasserXR supports both dynamic and static linking for your plugins:

|                              | Dynamic Linking        | Static Linking |
| ---------------------------- | ---------------------- | -------------- |
| Purpose                      | Development or modding | Release builds |
| Needs plugin loading         | Yes                    | No             |
| Allows hot-reloading         | Yes                    | No             |
| Needs plugin files at runtime | Yes                   | No             |

<Aside type="note" title="Hybrid Approach" icon="information">
  You can mix both approaches: statically link core functionality and
  dynamically load optional features. This also allows direct **modding
  support**. The engine treats them the same way once loaded.
</Aside>

### Static Linking

<Tabs syncKey="lang">
<TabItem label="Rust">

In Rust, it is recommended to use the following setup when you want a clean
development and release setup and you are working with other Rust
plugins.

Add the following to your `Cargo.toml`:

```toml
[dependencies]
your-plugin = { version = "<version>", optional = true }

[features]
default = []
static = ["dep:your-plugin"]
```

Then, in `main.rs`, add the following to make the static plugin symbols
available in the final binary:

```rust
// At the top of your file
#[cfg(feature = "static")]
use your_plugin as _;

// ...

// Where you load the plugins in your `main` function
#[cfg(not(feature = "static"))]
{
  scene
    .load_plugin("./path/to/plugin.so".to_owned())
    .expect("Failed to find plugin");
}
```

With this setup, when you run:

```bash
cargo run
```

it will use dynamic linking for development.

When you run:

```bash
cargo run --release --features static
```

it will statically link everything together and make a clean release build.

</TabItem>
</Tabs>
