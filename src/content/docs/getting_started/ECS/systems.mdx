Systems are where the logic lives in ECS. While
[components](/getting_started/ecs/components) hold data, systems process that
data and make things happen. A system runs every scene tick, finds
[entities](/getting_started/ecs/entities) with specific
[components](/getting_started/ecs/components), and performs operations on them.

## Creating Your First System

<Tabs syncKey="lang">
<TabItem label="Rust">

You can build your first system by defining the main system function:

```rust
#[system(entities = [["Health"]])]
fn health_drain(scene: &mut Scene, entities: Vec<Vec<Uuid>>) {
  for entity in &entities[0] {
    let Ok((health,)) =
      scene.query_mut::<(&mut usize,)>(*entity, "Health", &["current_health"])
    else {
      continue;
    };

    *health = health.saturating_sub(1);
  }
}
```

<Aside type="note" title="Name of the system" icon="information">

The name of the system is determined by the name of the system function. In this
case, it is `"health_drain"`.

</Aside>

This builds a system that receives all entities with a `Health` component in the
first query group. We then get the
`current_health` field from that component and decrement it by `1`, stopping at
`0`.

</TabItem>
</Tabs>

## Creating and Destroying Systems

<Tabs syncKey="lang">
<TabItem label="Rust">

To add or remove a system, use the following scene functions:

```rust
let _ = scene.add_system("health_drain".to_owned(), 100);
let _ = scene.remove_system("health_drain");
```

The `100` is the priority of the system. The higher the priority of a system,
the earlier it will be executed during a scene tick.

</TabItem>
</Tabs>

<Aside type="caution" title="More than one system" icon="warning">

Like with [components](/getting_started/ecs/components), a system can only be
instantiated once.

</Aside>

<Aside type="caution" title="Same priority" icon="warning">

When two systems have the same priority, then there is no guarantee which system
will run before the other.

</Aside>

## Attacher and Detacher

Sometimes you want to set up certain resources when the system is first attached
and destroy them again when the system is removed. For that, you can build an
attacher and a detacher.

<Tabs syncKey="lang">
<TabItem label="Rust">

```rust
#[attacher(health_drain)]
fn health_drain_attacher(_scene: &mut Scene) {
  // Do something
}

#[detacher(health_drain)]
fn health_drain_detacher(_scene: &mut Scene) {
  // Do something
}
```

</TabItem>
</Tabs>

## Scene Ticking

<Tabs syncKey="lang">
<TabItem label="Rust">

After initializing the `Scene`, your main program runs the main loop by calling
`scene.tick()`. This runs all systems in priority order from highest to lowest.

Normally, the function will return `true` if the `Scene` wants to continue
ticking.

If something tells the scene to exit, then `scene.tick()` will return `false`.
That signals to the main program that it should quit.

</TabItem>
</Tabs>
